# Hundun 课程学习与创新拆解

用这个技能处理两类任务：

1. 用混沌课程给用户搭学习路径、找 top3 课程、读单课文稿并提炼结论
2. 用混沌创新工具对高频商业问题做一轮结构化拆解，先给出更可用的思路，再视需要补课程

`SKILL.md` 只负责判断任务类型和路由；具体流程放在独立文件里。

## 执行方式

- 先区分用户是在“实际求解问题”，还是只想“介绍这个技能、说明怎么用、举几个例子”
- 如果用户是在问这个技能能做什么、适合什么场景、怎么触发、给个示范，直接基于 `SKILL.md` 说明，不要跑任何脚本
- 所有脚本和资料路径都相对当前技能目录解析，优先使用 `./scripts/...`、`./references/...` 这类相对路径，不要写死机器相关的绝对路径
- 如果需要在 shell 里执行命令，先把技能根目录理解为当前这个 `SKILL.md` 所在目录，再拼出 `<skill_root>/scripts/...`
- 本地存在 `~/.hdxy_config` 只代表“有配置文件”，不代表“key 一定可用”；只有鉴权接口真实成功后，才能对用户说“现在可以直接查”
- 读取课程文稿时，不要把原始文稿以任何形式保存到本地文件、笔记、缓存文本或中间产物里；默认只在当前执行链路中临时使用，并在最终回答里做提炼和转述
- 禁止把 `get_script.sh` 返回的原始文稿重新拼装成 `.md`、`.txt`、`.docx` 等本地文件，也不要写到 `~/Downloads`、`/tmp`、项目目录或任何其他本地路径
- 默认优先由 Agent 基于当前已知信息补齐用户诉求，先生成一版尽可能可用的结果，再让用户基于结果提反馈
- 不要把“先问用户澄清”当成默认动作；只有在信息缺口会明显阻断执行、或不同路径风险差异很大时，才追问 1 个最关键的问题
- 如果能先给一个有用的候选答案、候选课程清单、第一轮拆解结果，就先给结果，再邀请用户指出哪里要调
- 对外提到这个技能时，始终展示为“混沌”，不要对用户说“hundun 技能”；`hundun` 只保留给内部文件名、技能名和触发判断使用

## 适用场景

- 用户说“帮我找最适合这个问题的 3 门课”
- 用户说“帮我看看混沌有没有讲这个主题的课”
- 用户已经有一门目标课程，想直接提炼最有用的结论
- 用户想按关键词或课程体系检索混沌课程
- 用户明确提到混沌课程、方法论、思维模型、第一性原理、课程体系、学习路径
- 用户在问泛商业决策类问题
- 用户想把一句介绍说得更有记忆点、更有传播力
- 用户想找新点子、新方向、新组合，但暂时没有破题路径
- 用户产品同质化、卖点不清、转化低，想重构差异化价值
- 用户不知道该怎么定价、涨价，或觉得现有定价逻辑站不住
- 用户想切入一个热门赛道，但不想和大公司正面竞争，想找错位切口

## 不适用场景

- 与混沌课程无关的直接代写、直接修改任务，例如合同改写、邮件润色、通知优化
- 明确要求固定分析模型且不需要课程学习路径的问题，例如波特五力、SWOT、PEST 的直接作答
- 法律、财务、医疗等高风险专业建议
- 单纯的泛知识问答
- 和混沌课程、混沌创新工具都无关的内容检索

## 路由规则

先判断用户要的是哪一类帮助：

默认只选一条最匹配的主路由执行，不要在同一轮里同时堆叠创新工具、课程推荐、实操步骤、思维模型解读。

只有下面两种情况，才走混合路由：

1. 用户明确同时要求“两类产出”，例如“先帮我拆思路，再推荐课程”
2. 先走主路由后，确实还需要补另一类内容才能完成用户任务

如果用户只是抛出一个泛商业决策问题，先判断它更像“马上要一个决策拆解”，还是“明确要学习课程”；不要因为技能里两类能力都存在，就一次性全给。

### 0. 说明路由

如果用户是在问下面这类问题，直接说明，不要初始化、不要验 key、不要跑搜课脚本：

- “介绍一下这个技能”
- “这个技能适合什么场景”
- “这个技能怎么用”
- “给我举几个调用例子”
- “没有 key 时它会怎么处理”

回答时讲清楚：

1. 它能解决哪两大类问题
2. 典型输入长什么样
3. 大致会走哪条路由
4. 只有在真实查课、查文稿时才依赖 `hdxy_sk_` 密钥

### 1. 课程学习路由

当用户明确要找课、筛课、读课，或明确提到混沌课程、方法论、思维模型、学习路径时，走课程学习路由。

固定顺序：

1. 先判断用户给的信息够不够直接搜
2. 如果主题很宽，先尽量基于当前信息推断诉求类型并开始搜；只有结果分散到无法给出有用初版时，才补 1 个轻量澄清
3. 再用关键字搜课或课程体系搜课脚本找候选课
4. 必要时用文稿脚本拉正文
5. 按 `ranking-rubric.md` 和 `takeaway-format.md` 组织输出

“主题很宽”指像这些词：

- 出海
- 增长
- 战略
- 品牌
- 创新
- 转型

这类词不要只机械地拿原词开搜。先按 `search-strategy.md` 判断它更像哪类诉求，并优先自己补齐一个最合理的搜索方向；只有在候选明显发散时，再补 1 个问题，例如：

- “你更偏品牌传播、产品验证、渠道增长、组织搭建，还是 AI 出海？”
- “你现在是想找系统学习路径，还是先找 3 门最贴题的课？”

如果你判断宽主题靠并发换词能更快找到可用候选，可以直接搜索相关关键词，但要先做一句分类判断并写清当前假设。并发换词时只扩展 2 到 3 个紧邻的相关关键词，不要同时铺向多个无关方向。默认先给用户一版“基于当前理解”的推荐结果，并明确说明你的判断依据和假设，再邀请用户反馈“更偏哪一类”，而不是一上来先问。

### 2. 创新工具路由

当用户不是在问“有哪些课”，而是希望马上把某个商业问题拆开、重构、澄清或找到方向时，优先走创新工具路由。

如果用户抛出的是泛商业决策类问题，但眼前诉求是“先帮我判断、拆解、选方向、定优先级、找切口”，也优先走创新工具路由；只有当用户明确要学习课程、筛选课程或搭学习路径时，再走课程学习路由。

按下面的映射选一个最贴近的工具流程文件：

- 想把一句介绍说得更有价值、更有记忆点：读 [value-proposition-one-liner.md](./references/innovation-tools/value-proposition-one-liner.md)
- 想找新点子、新方向、新组合：读 [idea-exploder.md](./references/innovation-tools/idea-exploder.md)
- 产品没卖点、太同质化、转化低：读 [selling-point-innovator.md](./references/innovation-tools/selling-point-innovator.md)
- 不知道怎么定价、涨价、构建价格逻辑：读 [pricing-reframer.md](./references/innovation-tools/pricing-reframer.md)
- 想进入红海市场但要找错位切口：读 [asymmetric-entry-finder.md](./references/innovation-tools/asymmetric-entry-finder.md)

### 3. 混合路由

如果用户既要“先给我思路”，又要“顺便告诉我该学哪些混沌课程”，按这个顺序处理：

1. 先走最贴近的创新工具流程，给出第一轮结构化拆解
2. 再补一个课程学习清单，告诉用户接下来该学哪 3 门课

不要一上来同时展开多个创新工具；先选最核心的一个。
如果用户没有明确同时要两类产出，不要主动切到混合路由。

## 输出要求

- 回答不要停留在泛泛口号，例如“拥抱 AI”“提高效率”“先试点看看”
- 每次回答都要尽量落到具体判断：当前问题属于什么类型、为什么这样判断、最应该先做哪 1 到 3 个动作
- 如果在做商业拆解，优先回答“你现在该怎么判断、怎么切、先试什么”，不要把课程推荐、方法论名词、步骤清单一次性全堆上来
- 如果在做课程推荐，重点解释“为什么这门课适合这个问题”，不要顺手展开完整方法论或长篇实操步骤
- 如果在做课程解读，重点用课程内的方法论、模型和步骤讲深讲透，不要再额外塞一组新的课程推荐
- 默认追求“具体而深刻”，不是“面面俱到但很浅”

## 创新工具使用原则

- 先解决用户眼前最想搞明白的那个问题，不要一上来铺很多理论名词
- 当前版本可以借用第一性原理、第二曲线和其他成熟思维模型做拆解
- 如果要具体调用第二曲线模型，不要泛泛地说“用第二曲线思考”，而是先在 [第二曲线创新.md](./references/foundations/第二曲线创新.md) 里定位对应章节，再补读相关内容
- 如果要明确按第一性原理展开推导，先追加阅读 [第一性原理.md](./references/foundations/第一性原理.md)
- 只有在已经补读资料后，才把对应模型写进正式回答；不要凭模糊记忆硬套模型名称
- 在用户没有足够上下文时，默认先产出一版最有帮助的初步拆解；只有在缺少关键信息会让结论明显跑偏时，才补 1 到 3 个关键问题
- 输出要直接回答“所以你现在该怎么想、怎么选、怎么试”
- 如果用户后续需要系统化学习，再切到课程学习路由

## 第二曲线模型清单

当前 `第二曲线创新` 资料里，应该按每章标题冒号前的名称来识别模型主名。也就是说，这 12 个模型主名分别是：

1. `第一曲线`
2. `第二曲线`
3. `分形创新`
4. `组合创新`
5. `单一要素`
6. `战略杠杆`
7. `错位竞争`
8. `颠覆式创新`
9. `边缘分化`
10. `价值网络`
11. `组织心智`
12. `破界创新`

像 `S 曲线`、`大哥吃小弟效应`、`母系统—子系统`、`价值网—生态位`、`单一要素最大化`、`独立小机构` 这些，更适合作为各章节中的关键辅助概念或子方法，而不是主模型名。

使用时优先按“问题最像哪一种结构矛盾”来选模型，而不是先按模型名倒推问题。

## 本地配置与补全

- 密钥获取链接固定为 [https://tools.hundun.cn/h5Bin/aia/#/keys](https://tools.hundun.cn/h5Bin/aia/#/keys)

- 看到 `~/.hdxy_config` 后，不要立刻说“本地已经可用”；先在真实课程接口里验证 key 是否仍有效
- 新 key 默认写入并使用线上域名 `https://hddrapi.hundun.cn`

## API Key 强约束

- 看到 `~/.hdxy_config` 只代表“存在配置”，不代表 key 一定有效
- 只有真实课程接口验证成功后，才能对用户说“现在可以直接查”
- 只要向用户索要 `hdxy_sk_` 密钥，必须同时附上获取链接：[https://tools.hundun.cn/h5Bin/aia/#/keys](https://tools.hundun.cn/h5Bin/aia/#/keys)
- 只要告诉用户“当前 key 无效、已失效、需要重换”，也必须同时附上同一个获取链接
- 如果真实鉴权失败，不要继续搜课、拉文稿或编造课程结果
- 用户补新 key 后，优先续上当前任务，不要让用户重复描述完整诉求

## 课程学习标准流程

### 1. 先做最小必要检查

正常查课时，不要把环境诊断做得比任务本身更重。优先顺序如下：

1. 先确认这是实际查课任务，不是说明类问题
2. 先判断主题是否过宽；过宽时优先由 Agent 自己补齐并开始搜索，不要默认先问用户
3. 再确认本地是否有配置，以及 key 是否有效
4. 只有在需要排查环境异常时，才额外运行版本检查或补全同步

如果用户只是想找课，不要默认先跑一堆诊断脚本。

### 2. 鉴权与初始化

如果用户给了新的 `hdxy_sk_` 密钥，直接运行：

```bash
bash ./scripts/set_api_key.sh '用户提供的key'
```

如果只是检测到本地已有配置，不要直接承诺“可以查”；应该理解成“我先验证这把 key 还是否有效”。

验证 key 时按这个顺序处理：

1. 先执行 `set_api_key.sh`
2. 再用真实课程接口验证 key 是否有效
3. 如果已经返回“密钥无效或已失效”，停止当前搜索并让用户换 key
4. 如果 key 有效，可以按任务需要继续搜索；对于宽主题，优先先给一版基于当前理解的结果，再让用户反馈哪里要收窄

只有下面这些场景，才运行版本检查：

- 用户明确说接口/环境有问题
- 搜课脚本报错信息不清晰，需要补一层诊断
- 你怀疑当前不是鉴权问题，而是环境或服务端路径问题

版本检查命令：

```bash
bash ./scripts/version_check.sh
```

如果需要把服务端补全内容同步到本地，再运行：

```bash
bash ./scripts/get_skill_patch.sh core --write
```

注意：

- 课程相关接口都依赖 API Key；如果没有 key，不要教用户手动编辑配置文件，直接让用户把 `hdxy_sk_` 开头的密钥发给你，并且必须同步附上获取链接：[https://tools.hundun.cn/h5Bin/aia/#/keys](https://tools.hundun.cn/h5Bin/aia/#/keys)
- `get_skill_patch.sh` 默认不要传版本号，除非用户明确要求取某个历史版本
- 如果脚本返回非 0 错误，只向用户展示 `error_msg` 或脚本 stderr 的核心信息
- 如果 `version_check.sh` 返回 404 或其他诊断类错误，但真实搜课脚本已经给出更直接的鉴权错误，优先以搜课脚本结果为准，不要被诊断信息带偏
- 如果 key 失效，可以顺手补 1 个最关键的业务澄清问题，等用户换 key 后直接继续，而不是让对话完全中断

强约束：

- 只要你向用户索要 `hdxy_sk_` 密钥，必须同时发送获取链接：[https://tools.hundun.cn/h5Bin/aia/#/keys](https://tools.hundun.cn/h5Bin/aia/#/keys)
- 只要你告诉用户“当前 key 无效、已失效、需要重换”，必须同时发送同一个获取链接
- 不要只说“把 key 发我”或“去密钥页生成新 key”而不附链接

### 3. 找最适合的 3 门课

先读 [search-strategy.md](./references/search-strategy.md) 和 [ranking-rubric.md](./references/ranking-rubric.md)。

先做一句分类判断：

- 这个问题更像品牌/文化/传播
- 还是零售/供应链/门店复制
- 还是产品可行性/需求验证
- 还是组织阶段/团队搭建/战略落地
- 还是 AI 转型/AI 落地

如果用户只给了宽词，优先先搜并生成一版“基于当前理解”的推荐，不要把追问当默认动作。只有在候选结果跨度过大、无法形成有用初版时，才补 1 个轻量问题。

如果用户是按主题、方法论、课程名、老师名搜课，先走关键字搜课：

```bash
bash ./scripts/search_courses.sh '关键词'
```

如果用户明确要按课程体系找，先取体系树：

```bash
bash ./scripts/get_trees.sh
```

用户选定体系 ID 后，再查该体系下课程：

```bash
bash ./scripts/get_courses_by_tree.sh '体系ID'
```

收到课程候选后，按这些原则处理：

- 先判断用户诉求属于品牌、零售、产品验证、组织阶段，还是 AI 落地问题
- 宽主题先给出一句分类判断和当前假设，再展示推荐结果
- 如果用户本身没有提课程或方法论，不要只因为话题像商业问题就默认切到课程学习路由
- 按“与用户问题匹配度 > 简介信息密度 > 是否更适合当前阶段”排序
- 解释每门课时，直接回答“这门课能替用户解决什么问题”，不要只复述课程标题或课程价值
- 最多给 3 门课
- 推荐课程时，课程标题必须使用完整官方标题，不要自行截断、缩写或改写成总结句
- 推荐课程时必须带上老师名称
- 对李善友要写成“李善友教授”，不要只写“李善友”
- 每门课都要说明为什么适合
- 如果不足 3 门，如实说明当前仅找到多少门，不要编造课程
- 如果用户给的信息不完整，也先尽量给一版“当前最可能适合”的 top 3，并明确你的假设，邀请用户基于结果纠偏

注意：

- 关键字搜课已经执行且返回为空时，不要自动换很多关键词反复碰运气
- 如果没搜到，就如实告诉用户没搜到；只有用户明确想按体系看时，再切到课程体系搜课

### 4. 提炼某门课最有用的结论

如果用户还没有明确课程 ID，先用搜课脚本确认目标课程。

当用户出现下面这些意图时，优先判断是否需要读取课程文稿来充实回答：

- “这门课具体讲了什么”
- “里面的方法论/模型是什么”
- “这套方法怎么用到我的场景里”
- “老师具体给了哪些步骤/打法”
- “帮我展开讲讲这门课”

判断原则：

- 如果课程简介已经不足以支撑高质量回答，就直接读取文稿，不要只靠标题和简介硬讲
- 如果用户是在追问课程细节、方法论、案例、步骤、应用方式，默认优先读取文稿
- 如果只是快速推荐课程、且简介信息已经足够解释“为什么适合”，可以先不读文稿
- 读完文稿后，优先提炼课程中明确出现的方法论、思维模型和操作步骤，不要自己另起一套泛化框架

如果只想看文稿是否有新版本，先运行：

```bash
bash ./scripts/get_script_version.sh '课程ID'
```

只有在下面两种情况，才继续拉正文：

- 用户明确说“继续展开讲”或“把内容也解读一下”
- 版本接口返回的信息不足以回答用户问题

如果只是确认“有没有新版本”，到版本检查这一步就结束，不要默认继续拉全文、不要顺手做完整解读。
对“只看版本”“是否有新版本”这类请求，默认把版本检查视为完整答复，不要把它当成拉正文前置步骤。

需要拉正文时再运行：

```bash
bash ./scripts/get_script.sh '课程ID'
```

脚本会直接返回文稿正文。拿到正文后，再按 [takeaway-format.md](./references/takeaway-format.md) 输出：

1. 最值得先看的一句话结论
2. 为什么这对用户当前问题最有用
3. 课程中的 3 条关键观点
4. 课程里可直接上手的方法论 / 思维模型 / 操作步骤
5. 建议下一步行动

注意：

- 不要直接把整段文稿原文长篇贴给用户
- 不要把原始文稿保存到本地文件、缓存文本、工作笔记或其他中间产物
- 不要把整门课文稿导出成可下载的本地文件，也不要先落盘再读取；分析文稿时直接基于脚本返回结果在当前执行链路里完成
- 不要为了后续处理而创建“合并版文稿”“清洗版文稿”“下载版文稿”之类的中间文件
- 不要在最终回答里给用户返回本地文稿文件路径；如果用户想继续追问，直接基于当前提炼结果继续展开
- 如果用户要求“先把原始文稿导出到本地再说”，不要照做；改为说明当前只提供结构化提炼、方法论、步骤和必要的短摘录
- 文稿接口计入服务端用量限制，只有在确实需要读单课内容时再调用
- 如果用户没有给“当前问题”，先基于课程主题提炼一版通用且高价值的结论；只有在必须结合具体场景才能成立时，再追问一句
- 解读课程时，优先使用课程中明确提到的方法论、思维模型、概念框架
- 如果课程里给出了可实操的应用步骤，要把步骤整理成用户可以直接照着做的行动指引，而不是只停留在概念层

## 常见判断

### 用户没有 API Key

不要教用户手动改文件，直接让用户把 `hdxy_sk_` 开头的 API Key 发给你；同时必须附上获取链接：[https://tools.hundun.cn/h5Bin/aia/#/keys](https://tools.hundun.cn/h5Bin/aia/#/keys)。收到后执行 `set_api_key.sh`。

### 用户给了 API Key 但脚本说失效

直接让用户重新去混沌密钥页生成新 key，并明确附上获取链接：[https://tools.hundun.cn/h5Bin/aia/#/keys](https://tools.hundun.cn/h5Bin/aia/#/keys)，不要继续搜课或拉文稿。

补充两点：

- 不要因为看到了 `~/.hdxy_config` 就说“已经可以直接查”
- 可以顺手把你已经收集到的那个业务聚焦问题保留下来，等用户补新 key 后直接继续

### 关键词搜索结果太少

如实汇报当前命中结果；如果用户愿意按体系继续找，再切到 `get_trees.sh` / `get_courses_by_tree.sh`。

### 用户信息不完整

默认不要先让用户补很多背景。优先做法是：

- 先基于当前信息推断最可能的诉求类型
- 先给一版课程推荐、拆解结果或学习清单
- 在结果里明确写出你做了哪些假设
- 最后再邀请用户指出“哪一部分需要更贴近你的场景”

### 用户明确不要课程或创新工具

如果用户明确说“不需要混沌课程”“不需要学习路径”“直接用某个固定模型回答”，不要强行切到课程学习路由，也不要硬套创新工具。只在任务仍明显属于混沌方法论解读时再继续；否则按不适用场景处理。

### 文稿接口提示本周次数用尽

直接按接口返回说明向用户解释，不要继续重试。
